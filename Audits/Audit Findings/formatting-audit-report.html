Formatting & Visual Cohesion Audit

setd5syndrome.live · March 9, 2026 · 26 pages reviewed

Overall Verdict: The site reads as a thoughtfully built, parent-created resource with a strong visual identity. The navy/teal/amber palette, Lora + Inter font pairing, and collapsing header all work well. However, the page-by-page build process has introduced a layer of small inconsistencies — different variable names for the same color, two animation timing families, mixed link styles, and some pages missing shared components like the back-to-top button. None of these are show-stoppers, but tightening them up will make the site feel more polished and professionally cohesive. Below are 19 specific findings, grouped and prioritized.

1. Typography Consistency

T1: Two different body background color values Medium

index.html uses --bg: #fefdf9 while every other page uses #faf7f2 (via --cream or --bg). This creates a very subtle warm shift when navigating away from the homepage that undermines the feeling of a unified palette.

Affected: index.html

Fix: Change index.html's --bg to #faf7f2 to match the rest of the site.
T2: Two variable name families for the same background color Medium

Some pages use --bg and --ink (index, family-toolkit, setd5-at-a-glance, medical-handout-builder), while most others use --cream and --text. The values are the same (#faf7f2 / #2c2c2c), but having two naming conventions makes future updates error-prone and feels stitched together.

Affected: index.html, family-toolkit.html, setd5-at-a-glance.html, medical-handout-builder.html, iep-handout-builder.html

Fix: Standardize on one convention. Recommendation: --cream / --text since it's used by the majority of pages. But either works — just pick one and apply everywhere.
T3: about.html sets body font-family to Lora (serif) instead of Inter Low

The About page sets body { font-family: 'Lora', serif; } after an initial html, body { font-family: 'Inter', sans-serif; }. This means the body text on About renders in Lora while every other page renders in Inter. The effect is a slightly "softer" feel on About — which may be intentional for the personal narrative style, but it does break the pattern. Additionally, the html, body block also sets line-height: 1.65 which is then overridden by body { line-height: 1.75 }.

Affected: about.html

Fix: If the Lora body text on About is intentional, consider leaving it but adding a brief CSS comment noting the deliberate override. If not, remove the Lora override on body to match the rest of the site.
T4: Body line-height varies across pages Low

Body line-height is 1.65 on index.html, 1.7 on most content pages, and 1.75 on about.html. While the differences are small, they contribute to an uneven reading rhythm when moving between pages.

Affected: index.html (1.65), about.html (1.75), most others (1.7)

Fix: Standardize body line-height to 1.7 across all pages.

2. Spacing & Layout

S1: Inconsistent collapsed header padding Medium

The collapsed sticky header has at least four different padding values across pages:

8px 32px 10px (disability-discounts, index-adjacent pages) · 8px 32px (most pages) · 8px 18px 9px (about.html mobile) · 8px 18px (some mobile overrides)

This means the header height changes slightly when navigating between pages while scrolled, which creates a visual "jump."

Affected: all pages with sticky headers

Fix: Standardize collapsed header padding to 8px 32px 10px on desktop and 8px 18px 10px on mobile across all pages.
S2: Main content container approach differs between page families Medium

The site uses two different layout approaches. The "prose" pages (transition-to-adulthood, qualifying-for-services, helpful-links) put max-width: 860px on main with inline padding. The "band" pages (family-toolkit, about) use full-width sections with a nested .content-inner { max-width: 860px }. Both work, but they behave differently with alternating background bands — the prose-style pages can't do edge-to-edge alternating backgrounds.

Affected: transition-to-adulthood.html, qualifying-for-services.html vs. family-toolkit.html, about.html

Fix: This is an architectural pattern difference, not a bug. No immediate fix needed, but if you ever want to add alternating band backgrounds to the prose-style pages, you'll need to restructure their main container. Consider documenting the two patterns in a style guide comment.
S3: Some pages missing flex-direction: column and min-height: 100vh on body Low

Pages like disability-discounts-guide.html, helpful-links.html, share-your-story.html, and suggest-a-resource.html don't set min-height: 100vh; display: flex; flex-direction: column on the body. This means the footer may not stick to the bottom of the viewport on short pages.

Affected: disability-discounts-guide.html, helpful-links.html, share-your-story.html, suggest-a-resource.html, iep-handout-builder.html, medical-handout-builder.html

Fix: Add min-height: 100vh; display: flex; flex-direction: column to body on all pages, and ensure main { flex: 1; }.

3. Component Consistency

C1: Back-to-top button missing on 11 content pages High

The back-to-top button exists on 10 pages but is absent from: family-stories.html, helpful-links.html, setd5-at-a-glance.html, share-your-story.html, sources-references.html, suggest-a-resource.html, terms-of-use.html, transition-to-adulthood.html, and understanding-genetic-report.html. Several of these are long-scroll pages (helpful-links, sources-references, transition-to-adulthood) where the button would be especially useful.

Affected: 11+ pages (excluding builders/tools and internal-only pages)

Fix: Add the standard back-to-top button and scroll listener to all content pages. The button HTML + script pattern is already established.
C2: Two different link styling systems High

Some pages use text-decoration: underline via the blanket rule main a { text-decoration: underline } (index, family-toolkit, about, disability-discounts, setd5-at-a-glance), while others use border-bottom: 1px solid rgba(42,122,106,0.3) on the base a tag (transition-to-adulthood, qualifying-for-services, helpful-links, school-iep-guide, understanding-setd5-syndrome). The visual result is similar but not identical — the border-bottom approach gives a thinner, lighter underline that sits lower.

Affected: all pages

Fix: Pick one approach and apply it everywhere. Recommendation: the text-decoration: underline approach is more accessible and works better with wrapped text. The main a { text-decoration: underline } blanket rule is simpler to maintain.
C3: Header border-bottom only on about.html Medium

Only about.html has border-bottom: 4px solid #5a9186 on the header. No other page has this teal accent line below the header. On pages with the sub-navigation bar (family-toolkit, school-iep-guide, etc.), the teal nav bar serves a similar purpose, but pages without a sub-nav (transition-to-adulthood, helpful-links, etc.) have a bare transition from header to content.

Affected: about.html (has border) vs. all other non-nav pages (don't)

Fix: Either remove the border from about.html to match other pages, or add it to all pages without a sub-nav bar for visual consistency.
C4: Google Translate widget positioned inconsistently Medium

Most pages use position: absolute inside the header for the translate widget, but disability-discounts-guide.html, understanding-setd5-syndrome.html, and understanding-genetic-report.html use position: fixed. The about.html page puts its translate widget in the footer instead of the header. Fixed positioning means the widget stays visible while scrolling (which could overlap content), while absolute keeps it in the header only.

Affected: disability-discounts-guide.html, understanding-setd5-syndrome.html, understanding-genetic-report.html, about.html

Fix: Standardize on position: absolute inside the header. This matches most pages and avoids overlap issues.
C5: Footer is identical across all pages (good!), but index.html nav-bar uses inline styles Low

The footer markup and CSS is consistent across all 21+ content pages — great. However, the homepage navigation bar below the header uses entirely inline styles (style="background:#5a9186; padding:0...") rather than a class-based approach. The sub-nav on family-toolkit.html and school-iep-guide.html uses proper CSS classes (.section-nav). This makes the homepage nav harder to maintain and creates a disconnect.

Affected: index.html

Fix: Extract the inline styles on the homepage nav to a CSS class. Could use the existing .section-nav pattern or create a .site-nav class.

4. Visual Cohesion

V1: Two transition timing families (0.175s vs 0.25s) Medium

The header collapse animation uses 0.175s on some pages (about, family-toolkit, school-iep-guide, transition-to-adulthood, qualifying-for-services, disability-discounts, sources-references) and 0.25s on others (helpful-links, setd5-at-a-glance, family-stories, suggest-a-resource, terms-of-use, medical-terms, research). This means the header collapses at different speeds depending on which page you're on.

Affected: ~12 pages use 0.25s, ~9 use 0.175s

Fix: Standardize on one timing. Recommendation: 0.175s feels snappier and more polished. Apply to all pages.
V2: Header eyebrow link color varies Medium

The "← Back to..." eyebrow link and :visited state color varies: some pages use rgba(255,255,255,0.5), others 0.45, others 0.75. The visited state is especially inconsistent — some set it to 0.75 (brighter than default), some to 0.45 (dimmer). A few pages don't set :visited at all.

Affected: all subpages with eyebrow breadcrumbs

Fix: Standardize eyebrow link color to rgba(255,255,255,0.5) and :visited to rgba(255,255,255,0.5) (same as default, so visited doesn't look different).
V3: Index page uses a different header structure than all subpages Low

The homepage uses a plain <header> tag with non-sticky positioning, no eyebrow, and a larger visual treatment (centered h1, header-rule divider, .header-sub). All subpages use .site-header with sticky positioning, eyebrow breadcrumb, and the collapse-on-scroll behavior. This is likely intentional — the homepage serves as a "landing" page — but it means the header gradient overlay, font sizes, and layout are duplicated with slightly different implementations.

Affected: index.html

Fix: This is a design decision, not a bug. Keep as-is. Just be aware that any header changes need to be applied in two places: the index header and the .site-header pattern.
V4: H2 headings styled two different ways Low

Content pages like transition-to-adulthood, qualifying-for-services, and helpful-links style h2 with an amber left border (border-left: 4px solid var(--amber)). The about.html page styles h2 without a border, using a section-divider component instead. The family-toolkit page doesn't use h2 for section headings at all (uses .section-label divider pattern). These are all fine individually but represent three different heading treatments.

Affected: about.html (divider pattern), family-toolkit.html (label pattern), prose pages (border-left pattern)

Fix: Document the three patterns in a style guide comment. The amber border-left on prose pages, the section-divider on about/landing pages, and the label pattern on card-grid pages each serve a purpose. No change needed, but consistency within each page type should be maintained.

5. Readability & Accessibility

R1: Inline h3 styling on transition-to-adulthood.html Medium

The Financial Planning section uses inline styles for h3 headings: <h3 style="font-family: 'Inter'...">. This overrides the site's standard Lora serif h3 with Inter sans-serif and a smaller size (15px vs 22px elsewhere). It reads fine, but it's a one-off pattern that should be a class.

Affected: transition-to-adulthood.html (3 instances in Financial Planning section)

Fix: Create a CSS class like .subsection-title with these styles and replace the inline attributes.
R2: Title tags contain raw HTML entities on some pages Low

disability-discounts-guide.html and setd5-at-a-glance.html have <em> tags inside their <title> elements (e.g., <title>Disability <em>Discounts Guide</em></title>). Similarly, OG/Twitter title meta tags on these pages include <em> markup. The <em> tag doesn't render in title bars and will appear as literal text in search results and social shares.

Affected: disability-discounts-guide.html, setd5-at-a-glance.html (both <title> and og:title/twitter:title)

Fix: Remove <em> tags from <title> and <meta> content attributes. HTML tags don't render in these contexts.
R3: understanding-setd5-syndrome.html has malformed meta tag Low

Line 15 of understanding-setd5-syndrome.html is missing a closing > on the twitter:description meta tag. This could cause parsing issues in some browsers or social media crawlers.

Affected: understanding-setd5-syndrome.html

Fix: Add the missing > to close the meta tag.
R4: Some pages have extra blank lines in the <body> opening Cosmetic

Several pages (index.html, family-toolkit.html, helpful-links.html, etc.) have 3-4 blank lines between <body> and the first content element. This is harmless but looks unintentional in the source.

Affected: ~6 pages

Fix: Remove extra blank lines for cleaner source code.
R5: Missing canonical URL and meta description on 5 tool/internal pages Cosmetic

SETD5-Site-Audit-Prompts.html, element-reference.html, iep-handout-builder.html, medical-handout-builder.html, and style-guide.html lack rel="canonical" links. The handout builders would benefit from meta descriptions for search discoverability.

Affected: 5 pages (mostly internal/tool pages)

Fix: Add canonical URLs and meta descriptions to the two handout builder pages. The internal pages (audit-prompts, element-reference, style-guide) are less critical.

  1. Batch 1 — Quick wins (high impact, low effort): Fix the <em> tags in title/meta on disability-discounts-guide and setd5-at-a-glance. Fix the missing > on understanding-setd5-syndrome meta tag. Change index.html --bg from #fefdf9 to #faf7f2.
  2. Batch 2 — Link styling unification: Pick one link approach (recommend text-decoration: underline) and apply it across all pages. Remove conflicting border-bottom link styles.
  3. Batch 3 — Back-to-top button rollout: Add the back-to-top button + scroll script to the ~11 content pages that are missing it.
  4. Batch 4 — Header normalization: Standardize collapsed header padding, transition timing (0.175s), and eyebrow link colors across all pages.
  5. Batch 5 — Translate widget consistency: Change the 3 pages using position: fixed to position: absolute. Move about.html's translate widget from footer to header.
  6. Batch 6 — Body/layout normalization: Add min-height: 100vh; display: flex; flex-direction: column to the ~6 pages missing it. Standardize body line-height to 1.7.
  7. Batch 7 — Variable naming cleanup: Rename --bg/--ink to --cream/--text (or vice versa) on the 5 pages that use the minority convention.
  8. Batch 8 — Polish: Extract inline styles on index.html nav and transition-to-adulthood h3s to CSS classes. Remove extra blank lines. Add canonical/meta to builder pages.

How I Built This Website

Who this is for: This is a personal reference document, not published on the site. It explains how this website was built so I can maintain it, add pages, and pick up where I left off without having to relearn everything from scratch.

The Big Picture

This is a static website, meaning it is made entirely of plain HTML and CSS files with no database, no login system, and no content management software like WordPress. Every page is a single .html file that contains its own layout, styles, and content all in one place.

That approach was a deliberate choice. It means the site is fast, cheap to host, easy to back up, and I can open any file in a text editor and see exactly what it contains. There are no plugins to update, no server to manage, and nothing that can quietly break in the background.

The tradeoff is that changes have to be made file by file. There is no "update the header everywhere at once" button. But for a site this size, that is manageable.

Hosting

The site is hosted on GitHub Pages, which is a free service that turns a folder of files into a live website. The files live in a GitHub repository (a kind of cloud-synced folder with version history). Every time I push changes to the repository, the live site updates within a minute or two.

Custom domain The site lives at setd5syndrome.live. That domain was registered separately and connected to GitHub Pages using a file called CNAME in the root of the folder. The CNAME file contains nothing except the domain name on one line.

GitHub keeps a full history of every change ever made to the site. If something breaks, I can look back at what changed and undo it. This also means the entire website is backed up automatically at all times.

Design System

Every page uses the same set of colors, fonts, and spacing. These are defined as CSS variables at the top of each file and referenced throughout. The full visual reference is in element-reference.html.

Typography

Inter
Sans-serif font used for all body text, labels, buttons, and navigation. Clean and highly readable at small sizes. Used at weights 300, 400, 500, and 600.

The Role of AI in Building This

This website was built with the help of Claude, Anthropic's AI assistant. I have some technical experience in web development, but I am not a professional. I have a clear vision of what I wanted the site to be, and Claude helped me build it by writing HTML and CSS, troubleshooting problems, and translating my ideas into working code.

The workflow for most pages looked like this: I described what I wanted, Claude guided me in writing a draft, I previewed the page in a browser, gave feedback on what to change, and we iterated. For content, I supplied the personal knowledge; Claude helped me scour websites for research data and links to helpful resources, and helped me translate everything into clear, plain-language writing.

One of the most valuable parts of working with Claude was the brainstorming process. Rather than just executing instructions, Claude would ask clarifying questions that helped me think through what I actually wanted: Who is this page for? What is the single most important thing someone should take away from it? What do families typically struggle to understand about this topic? Those questions pushed me to articulate things I had not fully thought through, which made the final content much sharper. Claude also offered options and tradeoffs when there were multiple ways to approach something, which helped me make better decisions rather than just going with the first idea.

What I did vs. what Claude did I provided: the purpose and vision, all SETD5 Syndrome content and research knowledge, design preferences and feedback, and all editorial decisions. Claude provided: the HTML and CSS code, troubleshooting and debugging, structural suggestions, that I then reviewed and edited.

Using Cowork mode in the Claude desktop app made this workflow especially practical. It could read and write files directly in the folder, make targeted edits, and maintain context across a long working session.

How a Page Is Structured

Every inner page follows the same anatomy. When I make a new page I copy an existing one and replace the content.

<head>
Page title, favicon, meta description for Google, Open Graph tags for sharing, Google Fonts link, and the <style> block containing all CSS for this page.
.site-header
The dark navy header with the page title in Lora serif and a subtitle underneath. It sticks to the top of the screen and collapses as the user scrolls down (controlled by a small JavaScript snippet at the bottom of the file).
.container
The main content area. Centered, max 860px wide, with generous padding. All page content goes inside here.
.intro-box
A teal-bordered callout near the top that orients the reader and explains what this page is for.
h2 sections
Each major topic is an h2 heading with an amber left border, followed by one or more white .section cards containing the content.
<footer>
Dark navy footer with the "Made with love" line, navigation links, and the legal disclaimer.
<script>
A small block of JavaScript at the very end of the file handles the collapsing header on scroll. That is the only JavaScript on most pages.

How to Add a New Page

  1. Copy an existing page that is structurally similar to what I want. Rename the copy with a descriptive filename using hyphens, no spaces (e.g. new-topic.html).
  2. Update the <title> tag, the meta description, the Open Graph tags, and the canonical link in the <head>.
  3. Update the header: change the .header-title and .header-subtitle text.
  4. Replace all the content inside .container with the new page's content. Keep the same structural patterns: intro-box, h2 headings, .section cards.
  5. If the page should appear in site navigation or be linked from the home page, add a link to index.html and update any relevant nav menus in other pages.
  6. Add the new page URL to sitemap.xml.
  7. Save, commit the changes in GitHub Desktop (or via terminal), and push. The page will be live within a minute or two.

Maintenance Notes

Because styles are embedded in each page rather than in a shared stylesheet, a visual change (like a new accent color) has to be made individually in each file. For small changes, this is fine to do by hand. For large changes, Claude can apply the same edit across all files in the folder at once.

Before making any significant changes Make sure the GitHub repository is up to date (pull any pending changes first). That way there is always a clean rollback point if something goes wrong.

PDF files (the printable handouts) are generated separately using a PDF skill in Claude and saved into the folder. They are not auto-generated from the HTML pages.

The interactive builder tools (IEP Handout Builder, Medical Handout Builder, AI Prompt Builder) are all self-contained single-file HTML tools. All their logic is in JavaScript inside the file. They do not connect to any external service or database.

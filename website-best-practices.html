Building a Site Like This One

Best practices for efficiency, consistency, and sanity when building a static HTML website with AI assistance.

Who this is for: Anyone who wants to build a plain HTML/CSS resource website similar to this one. These practices are drawn from the experience of building setd5syndrome.live and are written for someone with some technical familiarity but not a professional development background.

Think in Three Phases

Before writing a single line of HTML, it helps to think of the project in three distinct phases. Mixing them together is the most common source of wasted time.

Phase 1: Foundation
  • Define the purpose and audience
  • List every page the site needs
  • Choose your color palette and fonts
  • Build one template page
  • Set up GitHub and hosting
Phase 2: Content
  • Write or gather all content first
  • Clone the template for each page
  • Add content before styling
  • Review for accuracy and tone
  • Add meta descriptions
Phase 3: Polish
  • Refine visual consistency
  • Test on mobile
  • Check all links
  • Update sitemap
  • Publish and verify live
The most common mistake Jumping straight into styling before the content is written. You will redesign everything once the real words are in place. Write first, style second.

Best Practices for Efficiency

  • 1
    Build a design system before building any pages

    Decide on your colors, fonts, and spacing before writing a single page. Define them as CSS custom properties (variables) at the top of your template file. Every decision you make upfront saves you from having to update twenty files later when you change your mind about a shade of blue.

    At minimum, define: background color, body text color, heading color, two accent colors, border color, and a muted text color. Give them meaningful names like --teal or --amber, not --color1.

    Time saver Ask Claude to generate a full CSS variable block from a description of the mood you want ("warm, trustworthy, approachable") and a few seed colors. Refine from there rather than starting from scratch.
  • 2
    Build one template page, then clone it for everything else

    Spend real time getting one page exactly right: the header, the layout, the typography, the footer, the card styles. Once that page looks and feels correct, every subsequent page starts as a copy of it. You only change the content, not the structure.

    This is the single biggest efficiency gain in the entire project. Without a solid template, every page becomes a fresh design problem.

    Time saver Keep your template file named something like _template.html so it is easy to find and never accidentally published.
  • 3
    Keep styles inside each HTML file

    For a site like this, embedding styles directly in each page (inside a <style> block in the <head>) is simpler to manage than a separate stylesheet. Each file is completely self-contained and nothing breaks if files get moved or renamed.

    The tradeoff is that global changes have to be made page by page. For a site with under 30 pages, this is manageable. For a larger site, a shared stylesheet would make more sense.

  • 4
    Write your content before you open a code editor

    The words on the page are what the site is for. Get them right first. Write your page content in a plain document or notes app, review it, edit it, and only then bring it into HTML. Trying to write and code simultaneously slows both tasks down.

    This is also where working with Claude is most effective: describe the page's purpose, share your raw notes and knowledge, and let Claude help you shape it into clear, organized prose that you then review and refine.

    Time saver Give Claude the messy version: your rough notes, copied research, half-formed thoughts. Ask it to turn that into a structured first draft. Editing a draft is much faster than writing from scratch.
  • 5
    Set up GitHub from day one

    Even before the first page is finished, initialize a GitHub repository and connect it to GitHub Pages. This means every change is versioned, every mistake is reversible, and the site is always backed up. It also means you can work on files locally and push to publish, which is a much cleaner workflow than editing files directly on a server.

    GitHub Desktop makes this approachable without needing to know command-line git. You can commit and push with two clicks.

    Time saver Name your repository after your domain name from the start. It avoids confusion later and makes the GitHub Pages URL predictable before the custom domain is connected.
  • 6
    Keep an element reference page

    As soon as you have established your visual language, build a reference page that shows every UI component: card styles, callout boxes, heading levels, button styles, color swatches, typography examples. This page is never published on the site but lives in the folder.

    When you return to the site after weeks away, this reference page tells you at a glance what patterns to use and what your design decisions were. It also serves as a spec you can hand to Claude: "make this match the style in my reference page."

  • 7
    Use consistent, descriptive file names

    Name every file in lowercase with hyphens between words: transition-to-adulthood.html, not TransitionToAdulthood.html or page3.html. Descriptive names become the URL and help both visitors and search engines understand what the page is about before clicking.

    Time saver Decide your file naming convention on day one and never break it. Renaming files later breaks links and requires updating every page that links to them.
  • 8
    Write meta descriptions for every page immediately

    The <meta name="description"> tag controls what appears under your page title in Google search results. Write it at the same time you write the page, while the content is fresh. Going back later to add descriptions to twenty pages is tedious and easy to forget.

    Keep descriptions between 140 and 160 characters. Be specific about what the page contains and who it is for.

  • 9
    Test in a real browser constantly, not just in preview

    Open the actual HTML file in Chrome or Safari as you build it. Preview panes in editors and AI tools do not always render things correctly, and they cannot show interactive behaviors like hover states, scrolling, or sticky headers. The browser is the ground truth.

    Also test on a phone before considering any page finished. Many layout issues only appear on narrow screens.

    Time saver Keep a browser window with the file open alongside your editor. After every meaningful change, switch to the browser and hit refresh. Catching problems immediately is far faster than debugging accumulated changes later.
  • 10
    Work with AI iteratively, not in one big request

    The most efficient way to use Claude is in short, focused cycles: ask for one thing, review it in the browser, give specific feedback, repeat. Large all-at-once requests ("build me this whole page") tend to produce results that need more correction than a series of smaller, directed steps.

    Be specific with feedback. "The heading feels too heavy" is harder to act on than "reduce the heading font size to 24px and change the weight from 600 to 400." The more precise the feedback, the faster the iteration.

    Time saver When something is working well, tell Claude explicitly: "keep everything about the card design, just change the border color." This prevents Claude from second-guessing things that are already correct.
  • 11
    Use AI for brainstorming, not just execution

    Claude is most useful before the coding starts. Describe the page you are planning and ask questions like: What sections should this page have? What would a first-time visitor want to know? What am I probably forgetting? The answers surface gaps and ideas that would otherwise only appear after the page is built and you realize something is missing.

    Claude can also push back usefully: if a structure you are proposing would confuse readers, it will often flag that before you invest time building it.

  • 12
    Keep a running list of what needs updating

    Maintain a simple text file or note with things you want to revisit: placeholder text that needs real content, pages that need a meta description, links you still need to find, sections that feel incomplete. Without this list, small unfinished items accumulate invisibly until the site feels rough in ways that are hard to pinpoint.

    Review the list at the start of every work session. Even completing one or two small items per session keeps the site moving forward steadily.

Common Time Traps to Avoid

The trap Why it costs time
Perfecting one page before moving on Design decisions that seem final always change once other pages exist. Build all pages to 80% first, then polish together.
Choosing colors by trial and error Color decisions should be made once, up front, with purpose. Changing a color after it is hardcoded across twenty pages is painful. Use variables from day one.
Starting from a blank page every time Always clone an existing page. Even a page that needs heavy editing starts faster than one built from scratch.
Not checking mobile until the end Retrofitting a desktop layout to work on mobile is harder than building responsively from the start. Check mobile after every major layout decision.
Vague feedback to Claude "Make it look better" produces unpredictable results. Specific feedback ("increase the line height to 1.7 and add 8px more padding inside the cards") produces exactly what you asked for.
Forgetting to commit before making big changes If you change many things at once and something breaks, it is hard to identify what caused the problem. Commit working versions frequently so you always have a clean rollback point.
Building features before content Elaborate interactive components on a page with placeholder text create a false sense of progress. Real content reveals real structural needs.

The Minimum You Need to Launch

It is easy to delay launching because the site does not feel "complete." Here is the actual minimum a site needs to go live and be useful:

Launch checklist A home page that explains what the site is and links to content. At least two or three content pages that deliver real value. A working domain or GitHub Pages URL. A meta description on every published page. No broken links. Readable on a phone.

Everything else, including more pages, refined visuals, a sitemap, Open Graph tags for social sharing, and additional features, can be added after launch. A live site with good content beats a perfect site that is never published.
